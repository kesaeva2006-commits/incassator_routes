# Руководство по стилю кода

## 1. Стандарты и принципы стиля кодирования

### 1.1. Форматирование

- Кодировка: UTF-8
- Отступы: 4 пробела (не табуляция)
- Максимальная длина строки: 79 символов
- Пустые строки: 2 между функциями, 1 между методами класса
- Пробелы вокруг операторов: `a = b + c`

### 1.2. Именование

| Элемент | Стиль | Пример |
|---------|-------|--------|
| Переменные, функции | `snake_case` | `travel_time`, `load_atms()` |
| Классы | `CamelCase` | `Atm`, `RouteBuilder` |
| Константы | `UPPER_SNAKE_CASE` | `WORKDAY_MINUTES` |
| Приватные атрибуты | `_snake_case` | `_current_in` |

Имена должны быть осмысленными. Запрещены: `a`, `b`, `x`, `temp`, `data`.

### 1.3. Документирование

- У каждой функции и класса — docstring с описанием параметров и возвращаемого значения
- Комментарии объясняют почему, а не что делает код

## 2. Логические операторы и условия

- Сложные условия выносить в отдельные методы с говорящими названиями
- Максимальная глубина вложенности — 3 уровня
- Использовать ранний возврат (`guard clauses`) вместо глубоких `if-else`

## 3. Принципы DRY и KISS

- DRY (Don't Repeat Yourself): повторяющийся код выносить в функции. Одинаковые блоки кода не копировать.
- KISS (Keep It Simple, Stupid): выбирать самые простые решения. Одна функция — одна задача.
- Константы: все повторяющиеся значения выносить в именованные константы. Магические числа запрещены.

## 4. Работа с базами данных

- Только параметризованные запросы (запрещены f-строки и конкатенация)
- Соединения закрывать в `finally` или через `with`
- Ошибки БД обрабатывать с выводом сообщения и откатом транзакции (`rollback`)
- Пароли и секреты не хранить в коде — использовать переменные окружения или `.env`

Руководство принято командой. Нарушения фиксируются при code review.
